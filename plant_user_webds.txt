## 1. PROJECT OVERVIEW

### 1.1 Purpose
Web registration system for **Duel Standby Guild** - a Yu-Gi-Oh! online gaming community. Members register, get verified by admin, and access community features.

### 1.2 Key Features
- ✅ Member registration with game profile validation
- ✅ Admin approval workflow (pending → active/rejected)
- ✅ IGN validation (must start with `[DS]`)
- ✅ Duplicate prevention (email, phone, game ID)
- ✅ Screenshot upload for verification
- ✅ Role-based access control (Member, Officer, Admin, Founder)
- ✅ Email notifications

### 1.3 User Roles
| Role | Permissions |
|------|-------------|
| **USER** | View profile, edit personal info | * ini adalah user web tp bukan member guild karena mereka yang bukan member juga bisa ikut tourmanet di duelist
| **MEMBER** | View profile, edit personal info | * ini adalah member guild yang sudah terverifikasi
| **OFFICER** | Member permissions + view all members | * ini adalah officer guild yang sudah terverifikasi
| **ADMIN** | Officer permissions + approve/reject members | * ini adalah admin guild yang sudah terverifikasi
| **FOUNDER** | All permissions + manage admin accounts | * ini adalah founder guild yang sudah terverifikasi

## 4. DATABASE SCHEMA

### 4.1 Prisma Schema (prisma/schema.prisma)

generator client {
  provider = "prisma-client-js"
}

datasource db {
  provider = "mysql"
  url      = env("DATABASE_URL")
}

enum Role {
  MEMBER
  OFFICER
  ADMIN
  FOUNDER
}

enum Status {
  PENDING
  ACTIVE
  REJECTED
  BANNED
}

enum GameType {
  DUEL_LINKS
  MASTER_DUEL
}

enum GuildStatus {
  SOLO_PLAYER
  LEFT_GUILD
  NEW_PLAYER
}

model User {
  id              BigInt      @id @default(autoincrement())
  fullName        String
  email           String      @unique
  password        String
  phoneWhatsapp   String?
  city            String?
  avatarUrl       String?
  status          Status      @default(PENDING)
  role            Role        @default(MEMBER)
  emailVerifiedAt DateTime?
  lastLoginAt     DateTime?
  createdAt       DateTime    @default(now())
  updatedAt       DateTime    @updatedAt
  deletedAt       DateTime?

  gameProfiles    GameProfile[]
  registrationLog RegistrationLog?
  adminLogs       AdminLog[]

  @@index([email])
  @@index([phoneWhatsapp])
  @@index([status])
  @@index([role])
}

model GameProfile {
  id            BigInt    @id @default(autoincrement())
  userId        BigInt
  user          User      @relation(fields: [userId], references: [id], onDelete: Cascade)
  gameType      GameType
  gameId        String
  ign           String
  screenshotUrl String?
  createdAt     DateTime  @default(now())
  updatedAt     DateTime  @updatedAt

  @@unique([userId, gameType])
  @@index([gameId])
  @@index([ign])
}

model RegistrationLog {
  id           BigInt      @id @default(autoincrement())
  userId       BigInt      @unique
  user         User        @relation(fields: [userId], references: [id], onDelete: Cascade)
  sourceInfo   String?
  prevGuild    String?
  guildStatus  GuildStatus
  socialMedia  Json?
  agreement    Boolean     @default(true)
  createdAt    DateTime    @default(now())

  @@index([userId])
}

model AdminLog {
  id        BigInt   @id @default(autoincrement())
  adminId   BigInt
  admin     User     @relation(fields: [adminId], references: [id])
  action    String
  targetId  BigInt?
  details   Json?
  createdAt DateTime @default(now())

  @@index([adminId])
  @@index([createdAt])
}

### 5.4 API Response Format

Success Response:

Error Response:

HTTP Status Codes:
- 200 - Success

- 500 - Server Error

## 6. AUTHENTICATION FLOW

### 6.1 Registration Flow
1. User Fill Form
2. Frontend Validate (Zod)
3. Backend Check Duplicates
4. Database Insert User
5. Backend Hash Password
6. Backend Trigger Email
7. Show Success to User

### 6.2 Login Flow
1. User enters email & password
2. Frontend validates input (Zod)
3. Backend finds user by email
4. Backend compares password (bcrypt)
5. Backend generates JWT/Session
6. Backend sets cookie
7. Frontend redirects to dashboard

### 6.3 Admin Approval Flow
1. Admin views pending members
2. Admin reviews registration data
3. Admin clicks Approve/Reject
4. Backend updates user status
5. Backend logs admin action
6. Backend sends email to user
7. Frontend refreshes list

## 7. VALIDATION RULES

### 7.1 Zod Schemas (src/lib/validators.ts)

import { z } from 'zod';

// IGN must start with [DS]
const IGN_REGEX = /^\[DS\].*/;

// Phone must be valid format
const PHONE_REGEX = /^\+?[0-9]{10,15}$/;

export const registerSchema = z.object({
  fullName: z.string().min(3, 'Nama lengkap minimal 3 karakter'),
  email: z.string().email('Email tidak valid'),
  password: z.string()
    .min(8, 'Password minimal 8 karakter')
    .regex(/[A-Za-z]/, 'Password harus mengandung huruf')
    .regex(/[0-9]/, 'Password harus mengandung angka'),
  phoneWhatsapp: z.string()
    .regex(PHONE_REGEX, 'Nomor WhatsApp tidak valid'),
  city: z.string().min(2, 'Kota harus diisi'),
  
  // Game Profiles
  duelLinksId: z.string().optional(),
  masterDuelId: z.string().optional(),
  duelLinksIgn: z.string().regex(IGN_REGEX, 'IGN wajib [DS] di depan').optional(),
  masterDuelIgn: z.string().regex(IGN_REGEX, 'IGN wajib [DS] di depan').optional(),
  
  // Registration Info
  sourceInfo: z.string().min(1, 'Sumber informasi harus diisi'),
  prevGuild: z.string().optional(),
  guildStatus: z.enum(['SOLO_PLAYER', 'LEFT_GUILD', 'NEW_PLAYER']),
  socialMedia: z.array(z.string()).min(1, 'Pilih minimal 1 sosial media'),
  agreement: z.literal(true, {
    errorMap: () => ({ message: 'Anda harus menyetujui pernyataan' })
  }),
});

export const loginSchema = z.object({
  email: z.string().email('Email tidak valid'),
  password: z.string().min(1, 'Password harus diisi'),
  rememberMe: z.boolean().optional(),
});

export const approveMemberSchema = z.object({
  status: z.enum(['ACTIVE', 'REJECTED']),
  reason: z.string().optional(),
});

// Type exports
export type RegisterInput = z.infer<typeof registerSchema>;
export type LoginInput = z.infer<typeof loginSchema>;
export type ApproveMemberInput = z.infer<typeof approveMemberSchema>;

## 8. CODING STANDARDS

### 8.1 TypeScript Rules
- Use strict mode ("strict": true in tsconfig.json)
- No any type (use unknown if necessary)
- Define types for all function parameters and returns
- Use interfaces for objects, types for unions
- Export types from src/types/index.ts

### 8.2 Naming Conventions
Files & Folders: kebab-case.tsx (Component files), kebab-case.ts (Utility files)
Variables & Functions: camelCase (Variables and functions), PascalCase (Components and classes)
Database: snake_case (Column names), PascalCase (Model names)

### 8.4 Error Handling
try {
  const result = await prisma.user.create({ ... });
  return NextResponse.json({ success: true, data: result });
} catch (error) {
  if (error instanceof ZodError) {
    return NextResponse.json(
      { success: false, errors: error.errors },
      { status: 400 }
    );
  }
  if (error instanceof Prisma.PrismaClientKnownRequestError) {
    return NextResponse.json(
      { success: false, message: 'Database error' },
      { status: 409 }
    );
  }
  console.error('Unexpected error:', error);
  return NextResponse.json(
    { success: false, message: 'Server error' },
    { status: 500 }
  );
}

| CSRF Protection | Use Next.js built-in protection |
| Rate Limiting | Max 3 requests per IP per hour |
